# Constitution - Ninjo（ニンジョ）

> プロジェクト全体に適用される不変の原則。すべての仕様・設計・実装はこの憲法に準拠すること。

## ミッション

> 勉強計画を立て、AIと一緒に「やり切る」を実現する。

## 不変の原則

### 1. 計画は立てて終わりじゃない

- 計画を立てることではなく、**計画を実行しきること**がこのアプリのゴール
- だから進捗管理・遅れ検知・軌道修正がコア機能になる
- 計画作成のハードルは極限まで下げる（教材名・ページ数・期限の3つだけ）

### 2. AIは「見てくれてる存在」

- AIは管理者ではなく、**一緒に走ってくれるコーチ**
- 計画の妥当性を正直にフィードバックする（甘やかさない）
- 遅れをネガティブに責めず、現実的な打ち手を提案する
- ユーザーの実績データを学習し、より精度の高い提案をする

### 3. メモは資産になる

- 学習中のメモはただの記録ではなく、**AIが分析する素材**
- 本を読み終えた時、メモから「何を学び、何が成長し、何を復習すべきか」が見える
- だからメモの入力体験は軽く、でも後から振り返れる形で保存する

### 4. シンプルさを守る

- 学習の単位はページ数。それ以上の複雑な単位を持ち込まない
- 1画面で「今日やること」が全て把握できること
- 設定は最小限。可処分時間の曜日設定だけで運用できる

### 5. データはユーザーのもの

- 学習メモ・進捗データは本人のものであり、適切に保護する
- データ削除リクエストには完全に対応する

## 技術原則

### T1. 仕様が先、コードは後（SDD）

- 新機能はまず仕様書を更新し、レビューを経てから実装する
- OpenAPI スキーマが BE/FE の契約。手動で型を書かない

### T2. 型で守る

- バックエンド: Go の構造体 + バリデーションタグで入出力を型定義
- フロントエンド: OpenAPI → TypeScript 型を自動生成（Orval）
- BE/FE間の型ズレを物理的に排除する

### T3. DDD + クリーンアーキテクチャを守る（4層構成）

- 依存の方向: handler → usecase → domain ← infra
- **Domain層**: エンティティ、値オブジェクト、リポジトリIF、サービスIF（最内側、依存なし）
- **Usecase層**: ビジネスロジック。Domainに依存
- **Handler層**: HTTPハンドラ、ミドルウェア、リクエスト/レスポンスDTO。Usecaseに依存
- **Infra層**: DB永続化（ent）、外部API実装。DomainのIFを実装（依存性逆転）
- ent の生成コードは Infra 層でのみ使用

### T4. セキュリティはデフォルト

- 認証・認可はすべてのエンドポイントに適用
- 環境変数で秘匿情報を管理。コードにシークレットを含めない
