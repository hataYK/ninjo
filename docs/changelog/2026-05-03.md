# 2026-05-03 技術選定・アーキテクチャ設計・プロジェクトルール策定

## 今日の進め方

SDDの方針に従い、コードを書く前に仕様・設計・ルールを固めることに集中した。
AIに調査・比較・提案をさせて、自分がレビュー・意思決定するスタイルで進めた。

---

## 1. 技術選定

### AIに依頼したこと
- Go/Echo vs Python/FastAPI の比較表を作成させた
- ORM比較（GORM vs ent）を調査させた
- フロントエンドの状態管理の選択肢を説明させた（そもそも状態管理とは何かから）
- axiosの脆弱性について指摘し、代替案を出させた

### 自分が判断したこと
- バックエンドはGoに決定（最初はFastAPIだったが変更）
- ORMはent（スキーマからコード自動生成がSDDの思想と合う）
- 状態管理はTanStack Query（サーバー）+ Jotai（クライアント）
- axiosは脆弱性ありのためネイティブfetchに変更
- APIクライアント生成はOrval（OpenAPI → TypeScript型 + hooks自動生成）

### 学び
- 技術選定はAIに比較表を作らせると判断しやすい
- 「なぜその技術か」を言語化しておくと後からブレない

## 2. アーキテクチャ設計

### AIに依頼したこと
- DDD + クリーンアーキテクチャのディレクトリ構成をWeb検索で調査させた
- Go界隈のOSSプロジェクト（bxcodec/go-clean-arch, Three Dots Labs等）の実例を集めさせた
- 自分が提案した3層構成（UI/Usecase/Datastore）に対するレビューを依頼

### 自分が判断したこと
- 最初は3層（UI / Usecase / Datastore）で設計した
- AIの調査結果を見て4層に変更を決定
  - `domain/` を独立層として分離（ドメインモデルがusecaseに従属しないようにする）
  - `ui/` → `handler/`（Go界隈の標準命名）
  - `datastore/` → `infra/`（DB以外の外部サービスも含むため）
- 依存方向: `handler → usecase → domain ← infra`

### 学び
- 自分で考えた構成をAIにレビューさせるのが効果的（最初から正解を聞くより理解が深まる）
- 命名は界隈の慣習に合わせるのが大事（独自命名は混乱のもと）
- ドメインモデルの独立性はDDDの根幹。層の配置が依存関係の誤解を生む

## 3. Git運用・プロジェクトルール策定

### AIに依頼したこと
- Conventional Commitsの仕様と実例を調査させた
- Git Flow / GitHub Flow / Trunk Based Development を比較させた
- ブランチ命名規則のベストプラクティスを調査させた
- SDDとの相性（specと実装を同じブランチにするか）を検討させた

### 自分が判断したこと
- コミット: Conventional Commits（英語サブジェクト + 日本語補足OK）
- ブランチ: GitHub Flow（個人開発にGit Flowは過剰）
- SDD対応: specと実装は同じfeatureブランチ、コミットのprefixで区別
- ルールの置き場所: CLAUDE.md + .claude/skills/ に配置
- スキルはClaude Codeが自動で実行すべき（手動呼び出しは手間）

### 学び
- ルールは「守らせる仕組み」とセットで決めるべき（CLAUDE.md, skills）
- 個人開発でもPR運用の癖をつけておくとチーム化がスムーズ
- スキルに切り出すとCLAUDE.mdがスリムになり、コンテキスト効率が上がる
